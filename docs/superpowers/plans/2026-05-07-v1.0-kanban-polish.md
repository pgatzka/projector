## Schema

**None.** v1.0 ships zero migrations. Status changes use the existing `PATCH /api/projects/{KEY}/issues/{N}` endpoint (which already emits a `status_changed` activity row from v0.5).

## Backend

**No new endpoints. No new services.** Kanban uses:
- `GET /api/projects/{KEY}/issues?size=100&...filters` — existing endpoint; client groups by status
- `PATCH /api/projects/{KEY}/issues/{N}` body `{status: "in_progress"}` — existing endpoint; emits activity automatically

**Known v1.0 limitation:** the issue list endpoint caps at `size=100`. For a project with >100 issues across all statuses, the board will only show the most recent 100. Document this in CLAUDE.md and the README; address in v1.1 if usage demands it (options: `?view=board&unpaged=true`, or per-column pagination, or move "done"/"cancelled" to a collapsible/load-more section).

### Toast implementation
Minimal, no library. Sketch:
```tsx
type Toast = { id: string; kind: "error" | "info"; message: string };
const ToastContext = createContext<{push: (t: Omit<Toast, "id">) => void} | null>(null);
// ToastProvider keeps a Toast[] in state, exposes push(), auto-dismisses after 5s,
// renders a portal div fixed bottom-right with stacked toasts, fade-out via CSS.
// useToast() throws if used outside the provider.
```

### Optimistic drag flow
1. User drops card on a different column.
2. `useBoardData` updates local state immediately (move card from source column to dest column, top of dest by `number desc` insertion).
3. Fire `patchIssue(key, number, {status: newStatus})`.
4. On 2xx: leave optimistic state in place; trigger react-query refetch in background to reconcile.
5. On 4xx/5xx: revert local state to pre-move snapshot; `toast.push({kind: "error", message: "Couldn't update <id> — <error>"})`.

### Sensor config (@dnd-kit)
- `PointerSensor` with `activationConstraint: { distance: 8 }` (avoids click-vs-drag confusion on the card body — the whole card opens the issue on click).
- `KeyboardSensor` with `coordinateGetter: sortableKeyboardCoordinates` for keyboard accessibility.
- Both sensors disabled below 768px (the mobile banner replaces the board entirely).

### Card click vs drag
- Whole card is the drag handle.
- Click (no drag distance exceeded) navigates to issue detail.
- The 8px activation distance keeps these distinct.

### KEY-N autolink details
- Pattern `\b[A-Z]{2,10}-\d+\b` (matches project key constraint: 2–10 uppercase letters).
- Skip if parent node type is `inlineCode` or `code` (don't link inside `` `ENG-7` `` or fenced code blocks).
- The link `href` is `/projects/{KEY}/issues/{N}` — internal SPA path, opens via React Router (use `<Link>` component override in react-markdown's `components` map for the `<a>` element).
- DON'T validate that the issue exists — that's a network round-trip per reference. Broken links 404 cleanly via the existing IssueDetail not-found path.

## Acceptance criteria per major behavior

1. **Board renders all open issues grouped by status** — fetching with `size=100` and existing filters; 5 columns in canonical order; correct counts per column.
2. **Drag changes status optimistically** — drop on different column → card moves immediately → PATCH fires in background → activity row appears (visible by reloading IssueDetail).
3. **Failed drag rolls back + toasts** — induce a 4xx (e.g. simulated by killing the network in DevTools) → card snaps back → toast appears bottom-right with the error message → toast auto-dismisses after 5s.
4. **Click vs drag is unambiguous** — short click on a card opens IssueDetail; pointer-down + 8px move starts a drag; no accidental opens during a drag attempt.
5. **Filters share state with list view** — set `?priority=high&label=<uuid>` on the board, switch to List view via toggle → the list view honors the same filters; switch back → board still filtered.
6. **KEY-N autolinks** — typing `ENG-7` in a comment renders a link; clicking navigates to `/projects/ENG/issues/7`; references inside `` `ENG-7` `` (inline code) or fenced code blocks are NOT linked.
7. **Tablet-or-larger only** — `<768px` shows the mobile banner with a clear message; the board itself doesn't render below that width; other pages still work (degraded layout is fine).
8. **MIT LICENSE in place** — file at repo root, attribution line present, no other licenses elsewhere.
9. **README is setup-runnable** — a fresh user who only reads README can `git clone && docker compose up` and reach a working `POST /api/setup` flow.
