# v0.5 — Comments + activity log + markdown

**Status:** planned (decisions locked 2026-05-07)
**Predecessor:** v0.4 (labels + filtering + FTS), tag `v0.4.0`
**Theme:** append-only comments per issue, an activity log capturing every save-driven field change, an interleaved chronological timeline on the issue page, and real markdown rendering replacing v0.3's `<pre>` placeholder.

This plan assumes everything in `2026-05-06-roadmap.md` and the locked decisions in v0.4 are still in effect. Re-read both immediately before executing.

---

## Decisions (locked via Q&A 2026-05-07)

| # | Decision | Choice |
|---|---|---|
| 1 | Markdown renderer | `react-markdown` + `remark-gfm` + `rehype-sanitize` (both descriptions + comments) |
| 2 | Timeline layout | single interleaved comments + activity stream below the issue, GitHub-style |
| 3 | Timeline order | oldest first, newest at bottom; composer at the bottom |
| 4 | Activity payload shape | snapshot before/after; entity references include human-readable name (tombstone-resilient) |
| 5 | Comments + FTS | separate `comment.search_tsv` (GIN-indexed); issue search query is union of `issue.search_tsv` match OR comment-issue match |
| 6 | Label rename / color change | does NOT generate activity rows on issues — only `label_added` / `label_removed` |
| 7 | Comment composer | plain textarea + "Preview" toggle button (single-pane) |
| 8 | Timeline pagination | single fetch — no pagination in v0.5 |
| 9 | due_date changes | emit `due_date_changed` activity rows |
| 10 | Issue creation | emit a synthetic `issue_created` activity row inside the create transaction |
| 11 | Comment body max | 10,000 chars (server-side `@Size` + frontend hint) |

### V22 — `comment` table

```sql
create table comment (
  id          uuid                     not null default uuidv7() primary key,
  created_at  timestamp with time zone not null default current_timestamp,
  created_by  varchar(64)              not null,
  updated_at  timestamp with time zone not null default current_timestamp,
  updated_by  varchar(64)              not null,
  version     integer                  not null default 0,
  issue_id    uuid                     not null references issue(id) on delete cascade,
  body_md     varchar(10000)           not null
);
```

Append-only at the API layer; the audit block stays for schema uniformity, but `updated_at`/`updated_by`/`version` will never change in normal use.

### V23 — set_updated_at trigger on comment

```sql
create trigger comment_set_updated_at
before update on comment
for each row execute function set_updated_at();
```

(Trigger never fires in normal use — comments aren't updated. Kept for uniformity.)

### V24 — generated tsvector on comment

```sql
alter table comment
add column search_tsv tsvector
generated always as (to_tsvector('english', coalesce(body_md, ''))) stored;
```

### V25 — GIN index on comment.search_tsv

```sql
create index ix_comment_search_tsv on comment using gin (search_tsv);
```

### V26 — `activity` table

```sql
create table activity (
  id          uuid                     not null default uuidv7() primary key,
  created_at  timestamp with time zone not null default current_timestamp,
  created_by  varchar(64)              not null,
  updated_at  timestamp with time zone not null default current_timestamp,
  updated_by  varchar(64)              not null,
  version     integer                  not null default 0,
  issue_id    uuid                     not null references issue(id) on delete cascade,
  action      activity_action          not null,
  payload     jsonb                    not null default '{}'::jsonb
);
```

### V27 — set_updated_at trigger on activity

```sql
create trigger activity_set_updated_at
before update on activity
for each row execute function set_updated_at();
```

## Activity payload shapes

| Action | payload jsonb |
|---|---|
| `issue_created` | `{}` (the row's created_at + created_by tell the story) |
| `status_changed` | `{"before": "todo", "after": "in_progress"}` |
| `priority_changed` | `{"before": "medium", "after": "high"}` |
| `due_date_changed` | `{"before": "2026-06-01", "after": null}` |
| `title_edited` | `{"before": "Old title", "after": "New title"}` |
| `description_edited` | `{"before": "...md...", "after": "...md..."}` |
| `label_added` | `{"labelId": "uuid", "labelName": "bug", "labelColor": "red"}` |
| `label_removed` | `{"labelId": "uuid", "labelName": "bug", "labelColor": "red"}` |

For `description_edited`, full before/after is stored. Acceptable for v0.5; if it bloats rows, switch to a diff representation in a later milestone.

## REST API

### Comments

| Method | Path | Body | Returns | Notes |
|---|---|---|---|---|
| POST | `/api/projects/{KEY}/issues/{N}/comments` | `{bodyMd}` | `CommentDto` (201) | append-only |
| GET | `/api/projects/{KEY}/issues/{N}/comments` | — | `CommentDto[]` | sorted `created_at asc`; convenience endpoint, even if the SPA mostly uses /timeline |

No PATCH, no DELETE. Comments are immutable post-creation.

### Timeline

| Method | Path | Returns | Notes |
|---|---|---|---|
| GET | `/api/projects/{KEY}/issues/{N}/timeline` | `TimelineDto` | merges + sorts comments and activity rows by `created_at asc` |

Response example:
```json
{
  "entries": [
    {"type":"activity","id":"...","createdAt":"...","createdBy":"user:...","action":"issue_created","payload":{}},
    {"type":"comment","id":"...","createdAt":"...","createdBy":"user:...","bodyMd":"first thoughts"},
    {"type":"activity","id":"...","createdAt":"...","createdBy":"user:...","action":"status_changed","payload":{"before":"todo","after":"in_progress"}}
  ],
  "total": 3
}
```

The two underlying queries are simple — service merges + sorts in Java (volume is low; no SQL UNION ALL needed).

### Issue list FTS extension

`GET /api/projects/{KEY}/issues?q=...` now matches an issue if EITHER its `search_tsv` matches the query OR any of its comments' `search_tsv` matches. No new query params; existing `q` shape unchanged.

### Markdown allowlist (rehype-sanitize)

Start from `defaultSchema`, then explicitly allow these GFM extensions:
- `<input type="checkbox" disabled />` (task list rendering)
- `<table>`, `<thead>`, `<tbody>`, `<tr>`, `<th>`, `<td>` with `align`
- `<del>`

Disallow: `<script>`, `<iframe>`, `<style>`, raw HTML attributes other than `href`. Limit `href` to `http`, `https`, and bare `mailto:` schemes.

Render every link with `target="_blank" rel="noopener noreferrer"` (via a react-markdown component override on `<a>`).

## Acceptance criteria per major behavior

1. **Comment create + render** — POST `/comments` with `{bodyMd}` returns 201; subsequent timeline GET includes the comment as the latest entry; markdown (paragraphs, code, tables, task lists) renders sanitized in IssueDetail.
2. **Append-only enforcement** — no PATCH or DELETE endpoint exists for comments; deleting an issue cascades comments away.
3. **Activity emission completeness** — for every save on an issue, one activity row per changed field; no rows for unchanged fields; create emits `issue_created`; idempotent label assign does not double-emit.
4. **Tombstone resilience** — assign label `bug/red` to issue → activity row exists; rename label to `defect/orange`, then delete it → fetching activity STILL shows "added bug (red)" because the snapshot was captured at emit time.
5. **FTS includes comments** — issue with no description-match but a comment containing the word is returned by `?q=<word>`.
6. **Timeline order** — entries sorted by `created_at asc`; composer at the bottom; new comment appears at the bottom on submit.
7. **Markdown sanitization** — pasting `<script>alert(1)</script>` in a description or comment renders as inert text; links open in a new tab with `rel="noopener noreferrer"`.
