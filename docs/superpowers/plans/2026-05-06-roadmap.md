## Cross-cutting decisions (locked in across all milestones)

These came out of the v0.2-onwards planning Q&A. They apply unless a later plan explicitly overrides them.

### Stack & infrastructure (carried over from v0.1)
- Java 21, Spring Boot 4.0.6, Maven single-module
- jOOQ + jOOQ codegen, generated POJOs only (no hand-written entity classes)
- Postgres 18 (`postgres:18-alpine`)
- Spring Security 7
- Single Docker container, jar built by Maven, frontend bundled via `frontend-maven-plugin`
- Layer packaging: `config/`, `audit/`, `data.repository/`, `data.service/`, `rest.controller/`, `rest.service/`, `rest.dto/`, `rest.exception/`
- Controllers never call repositories — controller → `rest.service` → `data.service` (or `data.repository`)
- Cucumber 7 for E2E (`*IT.java`), JUnit 5 for unit (`*Test.java`)
- Mandatory column block on every table (id uuid uuidv7 PK, created_at, created_by, updated_at, updated_by, version)
- `created_by`/`updated_by` are `varchar(64)` strings of form `"user:<uuid>"`, set automatically by `AuditRecordListener`
- All SQL lowercase, snake_case, table names singular

### Auth (set in v0.2)
- **Credentials:** email + password. `account.email` is unique, used as login identity.
- **Hashing:** bcrypt (Spring Security `BCryptPasswordEncoder`). `password_hash varchar(120)` (60-char bcrypt hash + headroom).
- **Account columns beyond audit block:** `email varchar(254)`, `password_hash varchar(120)`, `display_name varchar(100)`, `last_login_at timestamptz nullable`.
- **Sessions:** Spring Security default in-memory session, `JSESSIONID` cookie. **30-day idle timeout.** Lost on app restart — acceptable.
- **Setup gate:** `/api/setup` works only when `select count(*) from account = 0`. Returns `409 Conflict` afterwards.
- **CSRF:** enabled with double-submit cookie (`XSRF-TOKEN` cookie + `X-XSRF-TOKEN` header). Frontend wires up via fetch wrapper.
- **Logout:** `POST /api/logout` invalidates the session and clears the cookie. Returns 204.
- **Account creation post-setup:** none — single admin only in the v1.0 scope. Schema doesn't enforce this; the API does.

### Audit actor (set in v0.2)
- `ActorContext` reads from `SecurityContextHolder`'s `Authentication` principal. Returns `"user:" + accountId`.
- During `/api/setup`, the actor doesn't exist yet — `SetupService` constructs the actor string from the new account's UUID **before** the insert and uses a thread-local override for that one transaction.
- For Spring Boot startup events / Flyway / system tasks: `ActorContext.system()` returns `"system:bootstrap"` for the rare codepaths that need to insert without a logged-in user.

### Frontend (set in v0.2 onwards)
- **Routing:** project-scoped — `/setup`, `/login`, `/projects` (list), `/projects/{KEY}` (project home), `/projects/{KEY}/issues/{N}` (issue detail), `/projects/{KEY}/board` (kanban, v1.0).
- **401 handling:** redirect to `/login?next=<original-path>`; redirect back after successful login.
- **Protected routes:** all routes except `/setup` and `/login` require an active session — checked by a wrapper that reads `/api/me` once at boot and on 401.
- **Form save:** explicit Save button on edit pages. One PATCH per save click.
- **frontend-design skill** is invoked for any UI work (per CLAUDE.md).

### Data model (final shape after v1.0)

```
account            -- v0.2
  id, audit-block, email (unique), password_hash, display_name, last_login_at

project            -- v0.3
  id, audit-block, key (unique 2–10 A–Z), name, description, next_issue_number

issue              -- v0.3
  id, audit-block, project_id (FK), number (per-project), title, description_md,
  status (enum: backlog/todo/in_progress/done/cancelled),
  priority (enum: low/medium/high/urgent), due_date,
  search_tsv (generated column tsvector, GIN-indexed in v0.4)
  -- (project_id, number) is unique; user-facing id is f"{project.key}-{number}"

label              -- v0.4
  id, audit-block, project_id (FK), name (unique within project),
  color (palette enum: gray/red/orange/yellow/green/teal/blue/indigo/violet/pink/brown/slate)

issue_label        -- v0.4 (M:N join)
  issue_id (FK), label_id (FK), audit-block (yes — labels can be added/removed by anyone with access)
  -- composite PK (issue_id, label_id)

comment            -- v0.5
  id, audit-block, issue_id (FK), body_md
  -- append-only: no PATCH, no DELETE endpoints

activity           -- v0.5
  id, audit-block (created_at + created_by; never updated), issue_id (FK),
  action (enum: status_changed / priority_changed / label_added / label_removed /
          title_edited / description_edited),
  payload (jsonb)
```

Status, priority, label color, and activity action are Postgres-native enum types (one Flyway migration per enum: `V<n>__create_type_<name>.sql`).

### Delete semantics (set in v0.3)
- Hard delete everywhere. Cascade on FKs:
  - `delete project` → cascades to its issues, labels, activity
  - `delete issue` → cascades to its issue_label rows, comments, activity
  - `delete label` → cascades to issue_label rows
- No soft-delete columns. Activity log is the audit trail.

### Comments (set in v0.5)
- **Append-only.** No edit, no delete endpoints. Once posted, the comment is permanent.
- This means `updated_at` and `updated_by` will always equal `created_at` and `created_by` for comments — they're carried for schema uniformity, not because they vary.

### Activity log (set in v0.5)
- One activity row per change to: `status`, `priority`, label add/remove, `title`, `description_md`.
- Title/description edits are debounced — one row per **explicit save** (per the form-save model). Not per keystroke.
- `payload jsonb` carries before/after snapshots and any context (e.g. label name, since the label might be deleted later).

### Search (v0.4 + v0.5)
- v0.4: `issue.search_tsv` is a generated column over `title` + `description_md`, GIN-indexed. Search endpoint uses `websearch_to_tsquery`.
- v0.5: comments are added to the search story via a separate path — TBD when v0.5 plans land. Three options were discussed; the chosen approach is to keep v0.4 simple and re-evaluate v0.5 search when comments land.

### Pagination (set in v0.4)
- Offset/limit: `?page=N&size=M` (default size=50, max=100). Response wraps results with `{items: [], total: N, page: N, size: N}`.

### Kanban (v1.0)
- 5 columns = the 5 statuses. No custom column ordering.
- Drag-drop sets status only. Within-column ordering is `created_at desc` (no manual position).
- One PATCH per drop. Optimistic UI: card moves immediately, reverts on server error.

### What we are NOT building (in scope through v1.0)
- Multi-user accounts (single admin only)
- Issue assignee field
- File attachments
- Notifications (email, push, anything)
- Keyboard shortcuts (Linear-style)
- WIP limits
- Recurring issues
- Sub-tasks / parent-child issues
- Time tracking
- Edit history on comments
- Soft delete
- API versioning prefix
- Internationalisation

## Execution order

The milestones build on each other and **must be implemented in order**. Each milestone's plan ends with a `mvn clean verify` + smoke test gate; do not start the next milestone until the previous is green and reviewed.

Per milestone:
1. Read the milestone plan
2. Execute via `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans`
3. Run definition-of-done checks at the end
4. Manual smoke test of the new feature
5. Commit, optional push, then proceed to next milestone

## Plan revision policy

The v0.2 plan is the most detailed and most likely to be implemented exactly as written. The v0.3 plan is the next most concrete. Plans for v0.4 / v0.5 / v1.0 are based on choices that may want to be revisited once v0.2 / v0.3 are in your hands and you've used them. **Re-read and revise each plan immediately before executing it** — adjust based on what you learned from the previous milestone.
